1. Product Vision — What This Agent Must Deliver
This is not a simple crop calculator or a chatbot. This is a dedicated, always-running intelligence system that collects real data every night, learns from it, and answers one critical question for the farmer:
Core Question: If I grow this crop this season in Mandya — will I make profit or loss? What yield will I get? What price will I get when I sell? What is my risk?
The agent answers this for every crop, every season, with honesty — including high-risk crops like Tomato where prices swing from ₹2/kg to ₹50/kg. It never hallucinates. Every insight is cross-verified against data before it is shown. Every feature is tested for its impact before it is included.

4. Feature Engineering — Every Feature Tested, No Hallucination
This section defines every feature used in the ML models. Each feature has a purpose. Each feature must be tested for its actual impact on model accuracy before being included. If a feature does not improve the model — it is removed. No feature is added just because it seems logical. It must prove its value in data.
Anti-Hallucination Rule for Features: Before any feature is included in the final model, it must pass a feature importance test using XGBoost feature_importances_ or SHAP values. Features with importance below threshold are flagged and reviewed. Features that introduce data leakage are permanently banned. Every included feature is documented with its source, calculation method, and measured impact.

5. Anti-Hallucination System — Every Insight Cross-Verified
This is the most important engineering principle of this project. The agent must never show a prediction or insight that has not been verified. Hallucination in agriculture advice costs farmers money. The system has multiple layers of checks to prevent this.
5.1 The Four Verification Rules
Rule
Name
Description
Enforcement
Rule 1
Source Traceability
Every data point used in a prediction must have a source record in the database. If source is unknown, data is rejected before it enters the feature table.
validator.py — checks source_id field on every row
Rule 2
Feature Impact Proof
Every feature must demonstrate measurable improvement in model accuracy (RMSE reduction or SHAP value > threshold) before final inclusion. Intuition alone is not enough.
feature_selector.py — runs permutation importance tests
Rule 3
Prediction Range Check
Model output must fall within historically observed ranges. Paddy yield prediction of 200 qtl/acre is flagged immediately. Price prediction of ₹50,000/qtl for ragi is flagged.
anti_hallucination.py — crop-specific range validators
Rule 4
Cross-Source Consistency
When multiple data sources cover the same metric, they must agree within tolerance. If Open-Meteo and IMD rainfall differ by > 20% — neither is used until resolved.
validator.py — cross-source diff checker

5.3 Insight Verification Process
When the insight engine generates a text insight like 'Paddy yields are expected to be above average this Kharif', it must follow these steps before showing it on the dashboard:
Identify which features drove this insight — which columns had highest SHAP value for this prediction
Verify each driving feature has a clean data record with a traceable source in the database
Check if the same insight was generated in previous years and whether it proved correct — track record
Check if any conflicting data exists — if NDVI says good crop but rainfall was deficient, flag the conflict and show both signals
Only if all checks pass: display the insight on the dashboard with the supporting data visible
If any check fails: show 'Insufficient Data' instead of a guess
6. ML Models — Training, Evaluation & Walk-Forward Validation
6.1 Model 1 — Yield Prediction Model
Property
Specification
Algorithm
XGBoost Regressor (primary) with Random Forest as ensemble validator
Input Features
All features from Section 4 that pass feature selection — approximately 30-35 after selection
Target Variable
yield_per_acre in quintals — regression task
Hyperparameter Tuning
Optuna — automated Bayesian optimization. Run on each monthly retrain.
Training Data
Walk-forward: only years before prediction year. No future data ever used.
Evaluation Metric
RMSE (primary) + MAPE (farmer-understandable). Both reported after each validation.
Model Storage
Saved as crop_agent/models/yield_model_YYYYMM.pkl with metadata JSON
Retraining Trigger
Monthly scheduled retrain + triggered if MAPE drifts above 15% on live predictions
6.2 Model 2 — Price Forecasting Model
Property
Specification
Algorithm
Facebook Prophet for seasonal decomposition + LSTM for residual pattern capture
Input
Historical mandi prices (3+ years), arrivals volume, MSP, seasonal flags, export demand
Target
Price at harvest month — predict 3-6 months ahead from sowing date
Key Challenge
Tomato and other volatile crops have fat-tailed distributions. Use quantile regression for P10, P50, P90 predictions — not just mean.
Evaluation Metric
MAPE on held-out year prices. Also evaluate P10/P90 coverage — did the actual price fall within the predicted range?
Mandatory Output
Three price scenarios: Pessimistic (P10), Expected (P50), Optimistic (P90)
Model Storage
Saved as crop_agent/models/price_model_{crop}_YYYYMM.pkl — separate model per crop
6.3 Model 3 — Crop Profit Ranking Model
Property
Specification
Algorithm
LightGBM multi-output regressor — predicts expected profit for each crop simultaneously
Input
Season features (weather forecast, soil, irrigation) + output from Model 1 and Model 2 as stacked features
Output
Predicted net profit per acre for each crop — ranked from highest to lowest
Risk Adjustment
Final ranking adjusted by combined_risk_score — a high-profit but high-risk crop is ranked lower than a moderate-profit low-risk crop unless farmer explicitly wants high-risk options
Evaluation
Rank correlation (Spearman) between predicted ranking and actual profit ranking in validation year
Human Readable Output
Top 3 recommended crops with: yield, price, profit, risk level, confidence %, years profitable in last 5

10. Code Quality Standards — Production + Enterprise Grade
10.1 Every Line of Code Must Follow These Rules
Rule
Standard
Enforcement Tool
Typing
All functions must have type hints on every parameter and return value. No untyped functions.
mypy — fails CI if type errors found
Docstrings
Every class and every public method must have a Google-style docstring explaining what it does, what it takes, what it returns.
pydocstyle — checked in CI
No Magic Numbers
No raw numbers in code. All constants go in config/settings.py with descriptive names. Example: DROUGHT_THRESHOLD_PCT = 75, not just 75.
Code review — flagged in PR
Error Handling
Every external API call, file read, and database write must be wrapped in try-except. Specific exception types only — no bare except.
pylint — bare except rule
Logging
Every significant action logs: what happened, what data was involved, what the result was. Use structlog for structured JSON logs.
structlog — mandatory import in base classes
No Hardcoded Paths
All file paths and URLs go in settings.py or .env file. No hardcoded strings in logic code.
grep CI check — fails if path string in logic file
No Silent Failures
If a step fails and the pipeline continues, it must write to the anomaly_log. No step can fail silently.
Code review rule — every except block must log
Unit Tests
Every function that transforms data or makes a calculation must have a unit test. Minimum 80% code coverage.
pytest + coverage — fails CI if < 80%
No Data Leakage
Date filters in ML preparation code must be reviewed in every PR. Walk-forward assertions are non-removable.
walk_forward.py — assertion that raises RuntimeError if violated
Reproducibility
All ML training runs must log: random seed, package versions, feature list, training date range, evaluation metrics.
mlflow or custom experiment_log table
